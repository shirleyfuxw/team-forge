Concept explainer · team-forge extension

How team-forge turns a project description into a working agent team

team-forge is not a runtime skill kit. It's a factory. You feed it a project context — a codebase, a goal, a domain — and it walks you through a human-in-the-loop design loop that ends with concrete, project-specific agent files written into .claude/agents/. Every project gets its own bespoke team — but every team has the same role coverage: work, verify, advise.

TL;DR

team-forge has a layered design: the workflow, schema shape, and validation rules are fixed across every project (so the forge can enforce role-coverage and generate code); the values inside — agent names, phase lists, skill loadouts, escalation triggers — are dynamic per project. The HERC team is the prior art that proves the schema works.

01

The four-phase forge

Each phase has a clear input, output, and "who's driving" — the human, the AI, or both. Only the first three need human attention; phase 4 auto-generates.

PHASE 1 · human-led
Understand
Lead AI interviews the human about the project: domain, codebase, goal, constraints. Output: a 1-paragraph project brief.
PHASE 2 · human + AI
Decompose
Cut the goal into 2–5 milestones. Each one has a verifiable output + a human go/no-go decision point.
PHASE 3 · human + AI
Design team
Pick the agents per milestone. Forge enforces role coverage: every team needs work, verify, and advise agents — but the specific names + skills are project-shaped.
PHASE 4 · AI auto
Forge
Generate .claude/agents/*.md, scaffold skills, write hooks, create eval cases. Project is now equipped.
02

The three role types every forged team needs

team-forge's hard rule: a forged team without verify-role or advise-role agents is rejected at phase 3. The agents filling each role are project-specific — but the role coverage is universal.

● WORK
Does the domain task
Implementers, designers, builders. Produces the primary milestone output. Varies most by domain.
herc-implementer ui-designer model-trainer etl-builder
● VERIFY
Checks outputs before they propagate
Smoke-testers, reviewers, evidence-checkers. Catches hallucinations, regressions, future-leak. Output of every work-agent flows through verify.
herc-smoke-tester a11y-reviewer eval-runner load-tester
● ADVISE
Unblocks when work agents are stuck
Skeptics, librarians, domain experts. Called on-demand when a work-agent declares it's stuck. Holds prior context, challenges assumptions, escalates to human.
combiner-librarian combiner-skeptic paper-reader db-expert
03

The layered model — what's fixed, what's dynamic

team-forge gets its power from being concretely fixed where guarantees matter and dynamically open everywhere else. Four layers, top to bottom, each strictest at the top.

Layer
Fixed (universal across every team)
Dynamic (varies per project)
4
Validation rules
RULES ENGINE
Role coverage, phase coverage, comms closure, manifest integrity. Hard checks (refuse to forge) and soft warnings.
Nothing. Strictest layer. Change a rule and the brand promise breaks.
3
Per-entry shapes + enums
FIELD TYPES
Required fields on every roster[] entry (name, role, purpose, dispatch, active_phases, skills, model). Three enums: role (work / verify / advise), dispatch (always-on / per-phase / on-demand), channel.type (point-to-point / peer-visible / broadcast).
Optional fields per entry: modes, memory, shared_across_teams, extensions: {}.
2
Top-level YAML sections
SCHEMA SHAPE
Nine sections, names fixed: project, milestones, loop, roster, coverage, comms, escalation, memory, forge.
Contents of each section — values, list lengths, sub-structures.

PROCESS
Understand → Decompose → Design → Forge. Four phases, this order, every time.
How long each takes; questions asked in phase 1; granularity of milestones; revisions before phase 4.

06

How team-forge differs from a skill kit

team-forge generates artifacts; Superpowers ships them. The output of team-forge is your project's .claude/ directory; Superpowers' output is a session-time injection.

DimensionSuperpowers (skill kit)team-forge (factory)
OutputPre-built skills shared across all projectsProject-specific .claude/agents/*.md files + custom skills
PersonalizationNone — same kit everywhereEvery project gets a custom team shaped by domain + codebase
Role coverageNone enforced — depends on which skills you invokeEnforced: every team must have work, verify, and advise agents
SchemaN/A — skills are markdownLayered: fixed frame + dynamic content; concrete enums; manifest-backed
Human inputInstall once, use foreverRequired per project: brief → milestones → roster → comms
Runtime costWall-of-text SessionStart injectionZero — the generated agents stand alone, no team-forge dependency
MaintenanceUpstream owns the skillsEach project owns its forged team and can iterate
07

Glossary

Click any term to ask Claude to expand on it.

forge
The auto-generation step (phase 4): write actual agent files, skills, hooks, eval cases from the team design.
role coverage
The constraint that every forged team contains at least one work, one verify, and one advise agent. Enforced at phase 3.
work role
Agents that produce primary milestone output. Vary most by domain (implementer, designer, builder, trainer, …).
verify role
Agents that check work outputs before propagation. Domain-specific verifiers (smoke-test, a11y, eval, load).
advise role
Agents that unblock work agents on hard problems. Held in reserve, called on-demand (librarian, skeptic, expert).
project brief
The 1-paragraph output of phase 1. The forge's only persistent input from the human's domain knowledge.
milestone
A unit of work with a verifiable output and an explicit human go/no-go gate. Forge generates 2–5 per team.
comms protocol
Handoff schemas, visibility matrix, broadcast vs point-to-point, failure modes. Encodes how the three role types interact.
skill loadout
The subset of Claude Code skills a generated teammate is granted at dispatch time. Narrow on purpose.
eval scaffold
Auto-generated test cases per agent, dropped into agent_evals/. CLAUDE.md rule: every agent ships evals.
manifest
The forge's record at .claude/team-forge/<team>/manifest.json: which files it generated, design hash, shared agents used. Enables idempotent regeneration.
reference implementation
The combiner-team / combiner-peer-debate pattern in this repo is the prior art. team-forge generalizes it.
