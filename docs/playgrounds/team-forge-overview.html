What we build for the team team-forge

team-forge is the wiring that auto-generates a project-specific multi-agent team — roster, launcher skill, and a live dashboard — for any repo we own. Forge once per project; the team survives /resume.

TL;DR — Feed team-forge a project context. It runs a human-in-the-loop 4-phase design loop (brainstorm → plan → design → forge) and commits concrete files into the target repo's .claude/: agent definitions, a /<team>-team launcher skill, and a durable hub with a tracker + an HTML dashboard. The extension ships only the generic wiring; the domain-specific agents are forged into your project, owned by you. Click any card, phase, role, or callout to generate a focused prompt ↓

01 The 4-phase forge

A fixed pipeline. Phases 1–3 each end at a human approval gate; Phase 4 is autogen — it reads one annotated design.yaml and emits the whole team.

Each phase output is the next phase's input. The contract handed into Phase 4 is a single annotated design.yaml.

1Brainstormhuman + brainstormerhuman gate
Elicits the brief: other agents needed? what verification? what tracking? completion criteria? token budget? Ends with a 5-criterion self-review.
2Planhuman + plannerhuman gate
High-level milestones, each with hard dependencies, expected team size, a next-phase check, and the input/interface contract to the next milestone.
3Designparallel forge-design-agentshuman gate
Multiple agents review reciprocally (roster · comms+coverage · domain-fit), then reconcile into one annotated design.yaml: roster, asset discovery across plugins + reference libraries, comms patterns, rehydrate state.
4ForgeAI autoautogen
Reads design.yaml → emits agent .md files, the team-launcher skill, the hub (manifest, tracker, runtime, artifacts) and the initial rendered dashboard.html.

02 Five role types — coverage is enforced

Every forged roster must cover all five roles, or it's rejected at Phase 3. The tracker and monitor are load-bearing: their durable files are how a team survives /resume.

Work
Does the domain task. Mailbox to other workers + verifiers; receives assignments from lead.
ephemeral · rehydrated on resume
Verify
Checks outputs before they propagate. Reports verdicts to lead; status to tracker.
ephemeral · rehydrated
Advise
Unblocks work agents on hard problems. On-demand callable; reads memory + corpus.
ephemeral · rehydrated
Tracker
Aggregates state per the tracking: spec. Maintains status.json + appends events.
ephemeral agent · durable status.json
Monitor
Reads tracker + artifacts; rewrites dashboard.html per the dashboard-panels spec.
ephemeral · durable dashboard.html
Role coverage rule (hard): coverage is per-team-as-a-whole, not per-teammate. For one-shot domains, a single teammate may carry combined roles (e.g. combined_roles: [tracker, monitor]) — documented in design.yaml.

04 What ships in the extension

Only the generic, reusable wiring — installed once, used across every repo we own. Domain-specific agents are not shipped; they're forged per project.

Skills ×7
brainstorming · writing-plans · design · forge (phases 1–4) + rehydrate · tracker · monitor (runtime). Each phase skill self-reviews before the human gate.
Templates ×4
design.yaml · agent.md · team-launcher.md · dashboard.html — all logic-free {{VAR}} substitution.
Python renderer
tools/forge.py — deterministic alternative to the agent path. Both produce identical output.
Hook + manifests
Slim session-start hook, plugin manifests, and tests/ validation docs.

05 Surviving resume & mining prior art

team-forge bets on the experimental agent-teams primitive and designs around its limits — chiefly, that teammates evaporate on /resume.

Rehydrate protocol — load-bearing
On /<team>-team (fresh or after resume), the main session adopts the lead role, reads tracker/status.json + the current milestone's brainstorm/plan/artifacts, and respawns every teammate from disk. Single-writer rule keeps the durable hub race-free.
Reference libraries — prior art, not installed
During Phase-3 asset discovery, team-forge can mine external curated corpora (e.g. ECC: ~60 agents + ~250 skills) without vendoring or loading them as active skills. A pinned commit is fetched on demand, domain-filtered, and proposed as adapt candidates; a weekly workflow PRs pin bumps.
